# ENSF 533 Group 17 — Performance Evaluation

**Performance Evaluation of a Real-Time Multi-Radar Aerial Tracking Pipeline**

This repository contains the instrumentation code and raw measurement data used for the ENSF 533 final performance evaluation. It is a curated excerpt of a larger capstone project developed at the University of Calgary.

> **Note:** The full system — including radar protocol integration, hardware configuration, and the React frontend — is maintained in a private repository under capstone confidentiality requirements. This repo exposes only the components directly relevant to the performance evaluation: the instrumentation services, the load generator, the instrumented pipeline actors, and the collected CSV data.

## System Overview

The system under test is a `.NET 8` multi-radar aerial tracking pipeline that receives detections from two Sparrowhawk X-band radars and processes them through an event-driven actor pipeline:

1. **CentralTrackManager** — receives radar hits, runs track association (R-tree spatial index + Mahalanobis gating + Global Nearest Neighbour), and runs a Kalman filter update per track
2. **FusionTrackManager** — when a detection falls in the overlap zone of both radars, performs multi-lateration to estimate a 3D altitude
3. **SignalRHubActor** — broadcasts fused track state to a React frontend over a SignalR WebSocket connection

Performance is measured end-to-end per detection using `System.Diagnostics.Stopwatch` instrumentation embedded directly in each actor.

## Instrumentation

### Detection Latency (`LatencyMetricsService`)
A stopwatch is started at the entry to `HandleRadarHit` in `CentralTrackManager`:
- Split after track association → `AssociationMs`
- Split after Kalman filter update → `KalmanMs`
- `TotalMs = AssociationMs + KalmanMs` is the end-to-end detection latency

Written to: `metrics/detections-<phase>.csv`

### Fusion Latency (`LatencyMetricsService`)
A stopwatch wraps `HandleTrackUpdatedInOverlapZone` in `FusionTrackManager`.
Written to: `metrics/fusion-<phase>.csv`

### SignalR Dispatch Latency (`LatencyMetricsService`)
A stopwatch wraps the `SendAsync` call in `SignalRHubActor`.
Written to: `metrics/signalr-<phase>.csv`

### GC Pause Detection (`GcMetricsService`)
A background loop polls every 10 ms. If the elapsed time between polls exceeds 15 ms, the excess is recorded as a suspected GC pause along with Gen0/1/2 collection counts.
Written to: `metrics/gc-<phase>.csv`

### System Resource Monitoring (`SystemMetricsService`)
CPU utilisation (%), working-set memory (MB), GC allocated bytes, and thread count sampled once per second.
Written to: `metrics/system-<phase>.csv`

## Load Generator

Load is generated by `LoadGeneratorService` in the `radar-relay` service. It produces synthetic radar detections entirely in memory (no disk I/O) and forwards them to the AMS backend over a local SignalR WebSocket connection. All targets are placed within the radar overlap zone so every detection exercises the full pipeline including fusion.

### Configuration (`radar-relay/appsettings.json`)

```json
{
  "LoadGenerator": {
    "Enabled": true,
    "RatePerSecond": 100,
    "Pattern": "Steady",
    "DurationSeconds": 60,
    "TargetCount": 5,
    "ObservationsPerBatch": 1
  }
}
```

| Field | Description |
|---|---|
| `Enabled` | Set to `true` to use synthetic load; `false` uses live/file-based radar data |
| `RatePerSecond` | Total detections per second injected into the pipeline |
| `Pattern` | `Steady` (constant), `Spiked` (rate/2 normally, rate×10 for 2 s bursts every 10 s), `Decreased` (linear ramp to zero) |
| `DurationSeconds` | How long to run the load generator |
| `TargetCount` | Number of distinct synthetic targets to cycle through |
| `ObservationsPerBatch` | Detections emitted per batch tick (keep at 1 for normal use) |

## Experimental Phases

| Phase | Factor | Levels |
|---|---|---|
| 0 | Baseline (file-based, no synthetic load) | — |
| 1 | Concurrency sweep | 1, 10, 50, 100, 500, 600, 800, 1000, 1500 det/s |
| 2 | Target count | 1, 5, 20, 50 simultaneous targets |
| 3 | Input pattern | Steady, Spiked, Decreased |
| 4 | Run duration | 30 s, 5 min, 10 min, 30 min |

Each CSV filename follows the convention: `<metric>-<phase>-<condition>.csv`  
For example: `detections-p1-rate500.csv`, `fusion-p3-spiked.csv`, `gc-p1-rate1000.csv`
